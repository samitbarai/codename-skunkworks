## 1. Technology Decisions

### Why This Stack?

| Layer | Choice | Why |
|---|---|---|
| **Bundler/Framework** | Vite + React + TypeScript | Fastest DX, great ecosystem, SPA is fine for this product |
| **UI Components** | Base UI (`@base-ui/react`) | Truly unstyled — we own 100% of visual output. Accessible primitives without fighting overrides. |
| **Styling** | CSS Custom Properties (Design Tokens) | Primitive → Semantic → Component token layers. Works perfectly with Base UI's class-based styling API. No Tailwind specificity battles. |
| **Animations** | Framer Motion | Declarative, spring-based, works seamlessly with Base UI's `data-*` state attributes. |
| **Editor** | TipTap (ProseMirror) | Block-based, extensible, first-class Y.js collaboration via `@tiptap/extension-collaboration` |
| **CRDT / Real-time** | Y.js + Hocuspocus | Industry-standard CRDT. Hocuspocus is the TipTap-native WebSocket sync server. |
| **Backend / DB / Auth** | Convex | Real-time reactive queries, built-in auth (supports Google OAuth via OIDC), serverless actions for AI proxy, file storage. Eliminates need for separate WebSocket server + DB + API layer. |
| **AI Integration** | Convex Actions → Anthropic API | Serverless functions proxy LLM calls, stream responses back over Convex's real-time layer. |

## 3. Design Token System

Base UI components are unstyled. We control all visual output via **three layers of CSS custom properties**, applied using CSS Modules on each component.

```
Primitive tokens
      ↓  referenced by
Semantic tokens
      ↓  referenced by
Component tokens  ←  used inside *.module.css files
```

## 4. How Base UI + Tokens Work Together

Base UI components expose `data-*` state attributes (e.g., `data-popup-open`, `data-highlighted`, `data-disabled`). We style against those in CSS Modules using our semantic/component tokens:

```tsx
// components/ui/Button/Button.tsx
import * as BaseButton from '@base-ui/react/button';
import styles from './Button.module.css';

export function Button({ children, variant = 'default', ...props }) {
  return (
    <BaseButton.Root className={`${styles.root} ${styles[variant]}`} {...props}>
      {children}
    </BaseButton.Root>
  );
}
```

```css
/* components/ui/Button/Button.module.css */
.root {
  height: var(--btn-height-md);
```

```ts
// convex/schema.ts
import { defineSchema, defineTable } from "convex/server";
import { v } from "convex/values";

export default defineSchema({
  users: defineTable({
    name: v.string(),
    email: v.string(),
    avatarUrl: v.optional(v.string()),
    tokenIdentifier: v.string(),         // from Convex auth
  }).index("by_token", ["tokenIdentifier"]),

  workspaces: defineTable({
    name: v.string(),
    ownerId: v.id("users"),
  }),

  documents: defineTable({
    title: v.string(),
    workspaceId: v.id("workspaces"),
    parentId: v.optional(v.id("documents")), // for folder nesting
    isFolder: v.boolean(),
    yjsSnapshot: v.optional(v.bytes()),       // latest Y.js binary snapshot
    systemPrompt: v.optional(v.string()),     // per-doc AI persona
    createdBy: v.id("users"),
    updatedAt: v.number(),
  }).index("by_workspace", ["workspaceId"]),

  comments: defineTable({
    documentId: v.id("documents"),
    blockId: v.string(),                      // TipTap node ID
    selectedText: v.string(),
    authorId: v.id("users"),
    body: v.string(),
    resolved: v.boolean(),
    parentId: v.optional(v.id("comments")),   // threading
  }).index("by_document", ["documentId"]),
});
```

## 8. Y.js + Hocuspocus Setup

```ts
// src/lib/yjs.ts
import * as Y from 'yjs';
import { HocuspocusProvider } from '@hocuspocus/provider';

export function createDocProvider(documentId: string, token: string) {
  const ydoc = new Y.Doc();

  const provider = new HocuspocusProvider({
    url: import.meta.env.VITE_HOCUSPOCUS_URL,
    name: documentId,          // unique room per document
    document: ydoc,
    token,                     // Convex auth token passed for server auth
    onConnect: () => console.log('Connected to Hocuspocus'),
    onDisconnect: () => console.log('Disconnected'),
  });

  return { ydoc, provider };
}
```

```tsx
// src/components/editor/hooks/useCollaboration.ts
import { useEffect, useRef } from 'react';
import { createDocProvider } from '@/lib/yjs';
import { useAuthToken } from '@/features/auth/useAuth';

export function useCollaboration(documentId: string) {
  const token = useAuthToken();
  const ref = useRef<ReturnType<typeof createDocProvider> | null>(null);

  useEffect(() => {
    if (!token) return;
    ref.current = createDocProvider(documentId, token);
    return () => ref.current?.provider.destroy();
  }, [documentId, token]);

  return ref.current;
}
```

### 13.1 Gap Map: MVP vs Production

| Layer | MVP (Now) | Gap | Production Fix | When to act |
|---|---|---|---|---|
| **Real-time sync** | Hocuspocus (single node) | No horizontal scaling, single point of failure | Hocuspocus + Redis cluster OR migrate to Liveblocks | >500 concurrent editors |
| **Backend** | Convex | Vendor lock-in, 10s action timeout, unpredictable cost at scale | Postgres (Neon/Supabase) + custom WebSocket server | >50K MAU or cost alarm |
| **Y.js persistence** | Full binary snapshot per save | Snapshots grow unbounded, no compaction, no search indexability | Snapshot + incremental update log + periodic GC compaction | >1K documents or docs >10MB |
| **Auth** | Convex OIDC (Google only) | No RBAC, no token refresh strategy for long WS sessions, no multi-provider | Clerk (drop-in, handles RBAC + magic links + all providers) | Pre-launch or when 2nd auth provider is needed |

| **Full-text search** | None | No search across documents | Meilisearch or Typesense (self-hosted) or Algolia | When file tree has >50 docs |

| **AI rate limiting** | None | Free tier users can exhaust API budget instantly | Token bucket per user + tier quotas in Convex mutations | **Before AI ships** — P0 cost risk |

### 13.2 P0 Security Issues (Fix Before Any External Users)

These are not "scale" problems — they are **correctness and security** problems that must be fixed before sharing with anyone outside the team.

#### Authorization on Every Convex Function

Every `query` and `mutation` in Convex must check that the calling user has permission for the document/workspace being accessed. Currently the schema has no enforcement.

```ts
// convex/documents.ts — WRONG (no auth check)
export const getDocument = query({
  args: { documentId: v.id("documents") },
  handler: async (ctx, { documentId }) => {
    return await ctx.db.get(documentId);  // ❌ any authed user can read any doc
  },
});

// convex/documents.ts — CORRECT
export const getDocument = query({
  args: { documentId: v.id("documents") },
  handler: async (ctx, { documentId }) => {
    const identity = await ctx.auth.getUserIdentity();
    if (!identity) throw new Error("Unauthenticated");

    const doc = await ctx.db.get(documentId);
    if (!doc) throw new Error("Not found");

    // Check: is the caller a member of the workspace?
    const membership = await ctx.db
      .query("workspaceMembers")
      .withIndex("by_user_workspace", (q) =>
        q.eq("userId", identity.subject).eq("workspaceId", doc.workspaceId)
      )
      .first();

    if (!membership) throw new Error("Forbidden");
    return doc;
  },
});
```

Add a `workspaceMembers` table to the schema and gate every function.

#### AI Rate Limiting Before AI Ships

```ts
// convex/ai.ts — add rate limiting before any AI action
import { RateLimiter } from "@convex-dev/rate-limiter";

const rateLimiter = new RateLimiter(components.rateLimiter, {
  aiInvocations: { kind: "token bucket", rate: 10, period: 60_000, capacity: 20 },
  //              10 per minute, burst up to 20 — free tier
});

export const invokeAI = action({
  handler: async (ctx, args) => {
    const identity = await ctx.auth.getUserIdentity();
    await rateLimiter.limit(ctx, "aiInvocations", { key: identity.subject, throws: true });
    // ... rest of AI logic
  },
});
```

### 13.3 Y.js Snapshot Strategy: Do It Right From Day 1

The naive approach (store full snapshot on every change) will cause problems quickly. Build the correct pattern from the start:

```
┌──────────────────────────────────────────────────────────┐
│                    Y.js Persistence                       │
│                                                           │
│  Hocuspocus                                               │
│    ├─ onConnect   → load latest snapshot + update log     │
│    ├─ onChange    → append update to log (cheap)          │
│    └─ onDisconnect (last editor) → compact: new snapshot  │
│                                                           │
│  Convex Tables                                            │
│    ├─ yjsSnapshots  { documentId, snapshot: bytes }       │
│    └─ yjsUpdates    { documentId, update: bytes, seq }    │
│                                                           │
│  Compaction job (Convex scheduled action)                 │
│    └─ runs every hour: merge snapshot + updates → new     │
│       snapshot, delete old updates                        │
└──────────────────────────────────────────────────────────┘
```

This keeps individual writes tiny (update deltas are small) and snapshots compact (no unbounded growth).

### 13.4 Hocuspocus Scaling Path

```
Stage 1 (MVP):       Single Hocuspocus node on Railway/Fly.io
                     ↓ fine for <200 concurrent WS connections

Stage 2 (~500 CCU):  Hocuspocus + Redis adapter
                     Multiple Hocuspocus nodes behind a load balancer

Stage 3 (>5K CCU):   Evaluate Liveblocks (managed CRDT infra)
                     OR build custom Y.js sync server on top of
                     a persistent WebSocket framework (Durable Objects,
                     Soketi, or Elixir/Phoenix Channels)
```

### 13.5 Convex → Self-Hosted Migration Path

When Convex becomes a constraint (cost, query flexibility, or SOC 2 data residency requirements), the migration is **backend-only**. The frontend and editor don't change.

```
Convex Layer                    →  Self-Hosted Replacement
─────────────────────────────────────────────────────────
Convex DB (documents, users)   →  Neon (Postgres) + Drizzle ORM
Convex real-time queries       →  Server-Sent Events or WS from your API
Convex auth                    →  Clerk (handles Google + all providers)
Convex file storage            →  AWS S3 + CloudFront
Convex actions (AI proxy)      →  Node.js/Hono API routes on Railway/Fly.io
Convex scheduled functions     →  Inngest or pg-boss (Postgres job queue)
```

**Effort estimate:** 3–4 weeks for a two-person backend team. The frontend only changes the URL endpoints and client libraries — no component rewrites.
